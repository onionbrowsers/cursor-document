# 基于 XMind 的代码与浏览器测试

## 适用场景

- 用户手上有 XMind 格式的测试用例（如「真题考点·好题速递」），希望自动对照**当前代码**和**浏览器内页面**做一轮验证。
- 需要产出结构化、可留档的 **Markdown 测试报告**（含通过/待确认/需手动等状态）。

**触发词**：XMind 测试、测试用例跑一遍、根据 XMind 生成测试报告、代码+浏览器测试报告、搬山 XMind 测试。

---

## 用户必须提供的信息

在执行前，向用户确认并收集以下**必填**信息：

| 参数 | 说明 | 示例 |
|------|------|------|
| **文件 ID** | 搬山平台上的 XMind 文件 ID（用于下载） | `21820` |
| **文件输出目录** | 下载的 XMind 文件要保存到的**绝对路径**或**相对项目根路径** | `.cursor/docs/test-case/中价课` 或 `/Users/xxx/project/.cursor/docs/test-case/中价课` |
| **测试网址** | 要在浏览器中打开并参与对比的页面 URL | `https://test-m.gtgz.cn/precision-study/middle-course?channelId=34` |

## 用户可选提供的信息

| 参数 | 说明 | 何时需要 |
|------|------|----------|
| **登录账号** | 测试环境登录用户名/手机号等 | 页面需登录后才能看到完整功能时 |
| **登录密码** | 对应账号的密码 | 同上 |

若用户提供账号密码，则在打开测试网址后，引导用户完成登录（或根据页面是否有登录入口，提示用户手动登录后再继续）；未提供时，按未登录状态进行可见内容的对比与代码审查。

## 执行流程

### 1. 下载 XMind 测试用例

- 使用 **feishu-booking MCP** 的 `download_banshan_file` 工具。
- 参数：
  - `id`：用户提供的**文件 ID**。
  - `download_folder`：用户提供的**文件输出目录**（若为相对路径，先转换为项目内的绝对路径）。
- **不再需要** `session_id`（MCP 已支持自动获取）。
- 确认文件已下载到指定目录（如 `真题考点-好题速递.xmind`）。

### 2. 解析 XMind 得到测试用例结构

- XMind 本质为 Zip，内含 `content.xml`。
- 使用系统命令解压并读取 `content.xml`，或用脚本解析其中的 `xmap:sheet`、`xmap:topic` 层级。
- 将测试用例整理为**层级结构**（模块 → 子模块 → 用例步骤 → 预期结果），便于后续逐条对照。
- 若解析结果过大，可先输出一版「测试用例目录」给用户确认，再进入浏览器与代码对比。

### 3. 打开浏览器并访问测试网址

- 使用 **cursor-ide-browser**（或当前项目可用的浏览器 MCP）的 `browser_navigate`。
- 打开用户提供的**测试网址**。
- 若用户提供了**账号密码**且页面有登录入口，在能力范围内协助或明确提示用户完成登录；登录完成后再进行截图/快照。
- 使用 `browser_snapshot`、`browser_take_screenshot` 获取首屏或关键页面状态，用于与测试用例中的 UI/文案预期做对比。

### 4. 结合代码审查做逐条对照

- 根据 XMind 中的用例分类（如「首页金刚位」「题目列表」「题目解析页」「埋点」等），在**当前代码库**中定位相关文件（页面、组件、常量、接口调用等）。
- 对每条测试用例：
  - **能通过浏览器截图/快照验证的**：描述实际结果，并标注 ✅ 通过 / ⚠️ 待确认 / ❌ 不通过。
  - **仅能通过代码验证的**（如权限逻辑、埋点事件 ID、接口参数）：基于代码审查给出「实际结果」与状态。
  - **需要真实环境或人工操作的**（如 Apollo 动态配置、弱网、兼容性真机）：标注为 ⏳ 需手动测试，并简要说明验证思路。
- 涉及**埋点**时，对照 `constants/haboEventId.ts`（或项目内埋点定义）与测试用例中的事件 ID/参数，一致则通过，不一致则在报告中明确写出代码值与用例值。

### 5. 生成测试报告

- 输出一份 **Markdown 测试报告**，建议结构：
  - **标题**：与 XMind 主题或模块一致（如「真题考点·好题速递 测试报告」）。
  - **元信息**：测试环境（测试网址）、测试日期、测试方式（浏览器截图 + 代码审查）、是否登录等。
  - **分章节**：按 XMind 的一级/二级模块划分（如「一、本周好题模块改动」「二、真题速递页面」「三、题目解析页面」等）。
  - **表格**：每类用例用表格列出「#」「测试用例」「预期结果」「实际结果」「状态」「验证方式」。
  - **总结**：整体通过率、需关注问题（P0/P1/P2）、待手动验证项、埋点/兼容性等单独小结。
- 将报告保存到**用户指定的文件输出目录**下，文件名建议包含模块名与「测试报告」，例如：`真题考点-好题速递-测试报告.md`（若 XMind 文件名为 `真题考点-好题速递.xmind`）。

## 报告状态与优先级约定

| 状态 | 含义 |
|------|------|
| ✅ 通过 | 浏览器或代码与用例预期一致 |
| ⚠️ 待确认 / 差异 | 实现与用例有差异，需产品/后端确认（如配置项、文案） |
| ❌ 不一致 | 实现与用例明确不符（如埋点 ID 错误） |
| ⏳ 需手动测试 | 依赖真机、弱网、人工操作等，未在本流程中执行 |

优先级建议：**P1** 为阻塞或高优先级（如错误埋点、关键功能缺失），**P2** 为体验或配置类，**P3** 为优化或可选项。

## 注意事项

- **sessionId**：当前 MCP 已支持自动获取，无需向用户索要 sessionId。
- **文件输出目录**：必须是当前项目下有写权限的路径；若用户只给相对路径，则相对于当前项目根目录解析。
- **登录**：浏览器自动化下登录可能受验证码、多步跳转限制，若无法自动完成，明确说明「请用户手动登录后告知继续」。
- **XMind 结构**：不同 XMind 的层级深度可能不同，解析时保留「模块 → 子项 → 步骤 → 预期」的对应关系即可，报告中的章节可与 XMind 主题对应。
- **参考实现**：本仓库中已有示例报告 `.cursor/docs/test-case/中价课/真题考点-好题速递-测试报告.md`，生成报告时可参考其章节划分与表格格式。

## 简要输入示例

用户可以说：

- 「用 XMind 跑一遍测试：文件 ID 21820，输出到 `.cursor/docs/test-case/中价课`，网址 `https://test-m.gtgz.cn/precision-study/middle-course?channelId=34`。」
- 「根据搬山文件 21820 的测试用例，对 `https://test-m.gtgz.cn/precision-study/middle-recommend?channelId=34&gradeTermId=2` 做代码+浏览器测试，报告也放到 `.cursor/docs/test-case/中价课`。需要登录的话账号是 xxx 密码 xxx。」

你收到**文件 ID、输出目录、测试网址**后即可开始；若缺少任一项，先向用户索要再执行。
